## 📋 概述

系统中存在三种角色：
- **学生（student）**
- **教师（teacher）**
- **管理员（admin）**

不同角色查询"我的成绩"时，会有完全不同的行为！

---

## 👨‍🎓 学生查询"我的成绩"

### 行为

✅ **正常返回学生自己的成绩数据**

### 示例

**查询：** "我的成绩是多少"

**返回：**

```
✅ 查询成功！找到 16 条记录

💡 查询当前用户的所有成绩信息，包括姓名、课程名称、课程代码、总评成绩和学期。

您可以向下滚动查看完整数据，或点击下载按钮保存为Excel文件。

共 16 条记录
姓名    课程名称    课程代码    总评成绩    学期
陈丽    数据结构    CS201    62.00    2024-2025-1
陈丽    算法分析    CS202    71.00    2024-2025-1
...
```

### 安全机制

1. SQL生成：`SELECT * FROM grades WHERE student_id = ?`
2. WHERE条件：自动添加当前学生的student_id
3. 结果验证：验证返回的每条记录的student_id都是当前学生

## 👨‍🏫 教师查询"我的成绩"

### 行为

⚠️ **返回友好提示，告知教师账号不关联个人成绩**

### 示例

**查询：** "我的成绩是多少"

**返回：**

```
💡 温馨提示

教师账号不关联个人成绩数据。

如果您需要查询：
- 我教授的课程：请说"我的课表"或"我教的课程"
- 我教的学生的成绩：请说"我的学生的成绩"或"我班级的成绩"
- 某个学生的成绩：请说"[学生姓名]的成绩"或"[学号]的成绩"

示例：
- "我的学生的成绩" - 查看您教授的所有课程的学生成绩
- "数据结构的成绩" - 查看某门课程的学生成绩
- "陈丽的成绩" - 查看某个学生的成绩

如有疑问，请联系系统管理员。
```

### 推荐的正确查询方式

| 查询内容 | 正确的说法 | 返回结果 |
|---------|-----------|---------|
| 我教授的课程 | "我的课表" | 教师教授的课程列表 |
| 我教的学生的成绩 | "我的学生的成绩" | 教师教授课程的所有学生成绩 |
| 某门课程的成绩 | "数据结构的成绩" | 某门课程的学生成绩 |
| 某个学生的成绩 | "陈丽的成绩" | 某个学生的所有成绩 |

### 为什么这样设计？

1. **数据模型：** grades表没有teacher_id字段，成绩关联的是学生
2. **职责分离：** 教师的主要职责是教学，不是学习
3. **避免混淆：** 教师可能同时也是学生（进修），但应该用学生账号查询成绩

## 👨‍💼 管理员查询"我的成绩"

### 行为

⚠️ **返回友好提示，告知管理员账号不关联个人数据**

### 示例

**查询：** "我的成绩是多少"

**返回：**

```
💡 温馨提示

管理员账号主要用于系统管理操作，不关联个人成绩或学生数据。

如果您需要查询个人成绩或学生信息，请：
1. 使用您的教师账号或学生账号登录
2. 或通过管理后台的数据管理功能进行查询

如有疑问，请联系系统管理员。
```

### 为什么这样设计？

1. **职责分离：** 管理员账号用于系统管理，不是学习或教学
2. **数据安全：** 避免管理员误操作导致数据泄露
3. **审计追踪：** 管理员查看数据应该通过专门的管理后台，留下操作日志

### 管理员应该怎么查询数据？

管理员需要查看学生数据时，应该：
1. 使用专门的管理后台
2. 管理后台有完整的数据管理功能
3. 所有操作都有审计日志

## 📊 三种角色行为对比

| 角色 | 查询"我的成绩" | 返回结果 | 原因 |
|------|--------------|---------|------|
| **学生** | ✅ 支持 | 返回学生自己的成绩 | 成绩数据属于学生 |
| **教师** | ⚠️ 不支持 | 返回友好提示和建议 | 教师账号不关联个人成绩 |
| **管理员** | ⚠️ 不支持 | 返回友好提示和建议 | 管理员账号不关联个人数据 |

## 🔒 安全机制

### 学生查询的安全保障

```javascript
// 1. SQL权限修正
fixSqlPermissions(sql, user)
  ↓
学生查询grades表
  ↓
自动添加 WHERE student_id = 当前学生ID
  ↓
确保只能查询自己的成绩

// 2. 结果验证
validateQueryResults(results, user)
  ↓
检查每条记录的student_id
  ↓
如果student_id != 当前学生ID
  ↓
❌ 拒绝返回，记录安全警告
```

### 教师查询的安全保障

```javascript
// 1. 意图检测
检测到"我的成绩" + 教师账号
  ↓
识别为不合理查询
  ↓
返回友好提示

// 2. 引导正确用法
提示教师应该查询：
- "我的学生的成绩"
- "我的课表"
- "陈丽的成绩"
```

### 管理员查询的安全保障

```javascript
// 1. SQL权限修正
管理员查询grades/students/teachers表且无WHERE条件
  ↓
自动添加 WHERE 1 = 0
  ↓
返回空结果

// 2. 友好提示
管理员 + 空结果 + 敏感表查询
  ↓
返回友好提示
  ↓
引导使用管理后台

// 3. 结果验证（双重保险）
检查结果是否包含个人敏感数据
  ↓
如果包含
  ↓
❌ 拒绝返回，记录安全警告
```

## 💡 设计原则

### 1. 职责分离

每个角色有明确的职责：
- **学生：** 学习，查询自己的成绩
- **教师：** 教学，查询教授的课程和学生成绩
- **管理员：** 管理，通过专门的管理后台管理数据

### 2. 数据安全

- 学生只能看到自己的数据
- 教师只能看到自己教授的课程相关的数据
- 管理员通过管理后台查看数据，有完整的审计日志

### 3. 用户体验

- 不支持的操作不直接报错
- 返回友好的提示信息
- 引导用户使用正确的查询方式
- 提供具体的示例

### 4. 防御深度

- SQL权限修正
- 友好提示机制
- 结果验证
- 多层安全防护
