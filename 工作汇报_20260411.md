## 一、背景

用户希望在后台管理系统中上传文档 → 自动处理 → 前台智能体能实时检索问答。系统采用多文档 RAG（Retrieval-Augmented Generation）架构。

### 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│  前台 (Student Portal)                                       │
│  Port 3000: Node.js/Express 后端                              │
│  Port 5173: Vue.js 前端                                       │
│  RAG服务: ragService.js → retrievalEngine.js                  │
└─────────────────────────────────────────────────────────────┘
                              ↕ 同一份索引
┌─────────────────────────────────────────────────────────────┐
│  后台 (Admin Portal)                                        │
│  Port 3005: Node.js/Express 后端 (TypeScript)              │
│  Port 5176: Vue.js 前端                                     │
│  文档处理: incrementalPipeline.js → indexManager.js           │
│  RAG服务: unifiedRagService.js → unifiedIndexManager.js      │
└─────────────────────────────────────────────────────────────┘
                              ↓
                    文档库 (Shared Storage)
                    - 文档库/indexes/retrieval_index.json (统一索引)
                    - 相关文档/registry.json (注册表)
                    - 相关文档/doc_*/source/*.docx (源文件)
```

## 二、已修复的问题 ✅

### 1. 统一索引路径不一致 (Critical)
**问题**: 两个系统各自读写不同的索引文件
- `IndexManager` 写到 `E:\外包\教育系统智能体\retrieval_index.json`
- `UnifiedIndexManager` 读 `E:\外包\教育系统智能体\文档库\indexes\retrieval_index.json`
- 前台 RAG 读根目录 `retrieval_index.json`

**修复**:
- `IncrementalPipeline` 改用 `unifiedIndexManager.addDocument()` 作为唯一写入点
- 前台 `ragService.js` 优先读取统一索引路径
- 数据已同步到统一索引位置

### 2. 事件订阅缺失 (Critical)
**问题**: `UnifiedIndexManager` 只订阅了 `INDEX_UPDATED/INDEX_REBUILT/INDEX_CLEARED`，漏掉了 `DOCUMENT_INDEXED` 事件

**修复**: 新增订阅 `DOCUMENT_INDEXED` 和 `DOCUMENT_DELETED` 事件

### 3. stop.bat 脚本编码崩溃 (Medium)
**问题**:
- 窗口标题中文乱码：`taskkill /FI "WINDOWTITLE eq 后台后端-3005*"` 被解析成 `/F:/`
- `快速启动.bat` 和 `stop.bat` 标题不匹配

**修复**: 改用端口号 + `PowerShell Stop-Process`，彻底解决编码问题

### 4. getDocumentChunks 路径错误 (Critical)
**问题**: `getDocumentChunks` 读取两个错误路径：
- `文档提取/文档名/retrieval_index.json` → 目录可能为空
- `IndexManager.loadIndex()` → 指向错误位置

**修复**:
- 改用 `unifiedIndexManager` 直接读取内存缓存
- 支持注册表ID与统一索引ID不一致时的文档名匹配
- 修复 `unifiedIndexManager.index` 未初始化导致的 null 错误

### 5. 删除不彻底 (Critical)
**问题**: 删除文档时只删了旧 `IndexManager`，统一索引没清理

**修复**:
- `IncrementalPipeline.deleteDocument()` 改用 `unifiedIndexManager.removeDocument()`
- 新增 `DELETE /api/rag/documents/:documentId` 端点

### 6. 增量流水线重建逻辑 (Medium)
**修复**: `rebuildDocument()` 同时清理统一索引和旧索引

## 三、仍存在的严重问题 🔴

### 1. Embedding 缓存加载不完整 (Critical)
**现象**:
```
统一索引文件: 268 chunks，全部有 embedding
内存缓存: 只有 1 条记录
```

**原因分析**: `unifiedIndexManager.loadCache()` 读取的是 `文档库/indexes/embedding_cache.json`，但这个文件可能不存在或格式不对。实际的 embedding 存在每个 chunk 的 `embedding` 字段里，不在单独的缓存文件。

**影响**: RAG 检索时内存中没有预加载 embedding，每次都要从 chunk 字段读取，性能差。

### 2. RAG 检索相似度极低 (Critical)
**现象**: 宠物领养系统文档有 16 个 chunks，全部有 embedding，但检索"宠物领养系统有哪些功能"时：
- 宠物文档匹配分数: **36.7%** (极低)
- 实际回答: 全是学生手册的内容
- 学生手册分数: 40.2% (反而比宠物文档高!)

**原因分析**:
1. 问题 embedding 和 chunks embedding 的相似度计算可能有问题
2. `UnifiedRagService.ask()` 中 `minScore = 0.3` 默认值太高，导致宠物文档被过滤
3. 可能是 embedding 模型问题或向量维度不匹配

### 3. 文档 ID 体系完全混乱 (Critical)
**现象**:

| 文档 | 注册表 ID | 统一索引 ID |
|------|-----------|-------------|
| 学生手册 | `doc_1775891409...` | `doc_student_handbook` |
| 宠物领养系统 | `doc_1775891974...` | `doc_1775900105...` |
| TCP/IP 协议 | ❌ 不存在 | `doc_1775896616...` |

**根因**:
- 注册表由 `DocumentRegistry` 管理，ID = `doc_时间戳_随机`
- 统一索引由 `UnifiedIndexManager` 管理，ID = `doc_时间戳_随机`（上传时新生成）
- 两边 ID 生成逻辑完全独立，没有任何关联

**影响**:
- 无法通过注册表 ID 找到统一索引中的文档
- 删除时可能只删一边
- 前后台无法对齐

### 4. 多个 registry.json 文件 (Critical)
**现象**: 项目中有三个 registry.json：
```
E:\外包\教育系统智能体\后台管理系统\相关文档\registry.json   ← 后台实际用这个 (空)
E:\外包\教育系统智能体\文档库\registry.json               ← 空
E:\外包\教育系统智能体\相关文档\registry.json               ← 前台/旧系统用这个 (有数据)
```

**影响**: 谁是 Single Source of Truth？

### 5. 注册表数据重启丢失 (Medium)
**现象**: 后台重启后，注册表变成空数组或只有部分数据。

**原因**: 注册表文件路径配置可能有问题，后台每次重启读取不到正确位置。

### 6. 前后台 RAG 服务架构割裂 (Critical)
**现象**:
- 后台: `unifiedRagService.js` → `unifiedIndexManager.js` → `文档库/indexes/`
- 前台: `ragService.js` → `retrievalEngine.js` → 各自读各自的

**影响**: 两边是独立的 RAG 实现，共用索引文件但检索逻辑不同，可能导致不一致。

### 7. ZHIPUAI_API_KEY 环境变量未正确传递 (Medium)
**现象**: 很多测试因 401 Unauthorized 失败，无法生成 embedding。

**原因**: 环境变量没有在所有服务启动时设置。

## 四、数据流向分析

### 当前上传→处理→检索的完整流程

```
用户上传文件
    ↓
[POST /api/documents/upload]
    ↓
DocumentRegistry.registerDocument()  ← 生成 doc_时间戳_ID
    ↓ 写入 registry.json (E:\外包\教育系统智能体\相关文档\)
    ↓ 复制文件到 相关文档/doc_xxx/source/
    ↓
[POST /api/documents/:id/process]
    ↓
IncrementalPipeline.processDocument()
    ↓
    ├─ Python extract_docx.py → extracted/content.json
    ├─ Python create_chunks_new.py → chunks/chunks.json
    ├─ 向量化每个 chunk (ZHIPU API)
    └─ unifiedIndexManager.addDocument()  ← 写入 文档库/indexes/retrieval_index.json
        ↓
    eventBus.notifyIndexed()  ← 触发 UnifiedIndexManager 重新加载
    ↓
[GET /api/rag/stats] → unifiedRagService → unifiedIndexManager  ✅ 工作正常
    ↓
[GET /api/rag/answer] → unifiedRagService
    ├─ 生成问题 embedding (ZHIPU API)
    ├─ unifiedIndexManager.retrieve()  ← 从内存缓存检索
    │   └─ cosineSimilarity() 计算相似度
    └─ 生成回答 (ZHIPU API)  ⚠️ 检索分数过低，返回错误内容
```

### 前台学生端流程（割裂状态）

```
前台学生提问 (Port 5173)
    ↓
[POST /api/rag/answer] → 前台后端 (Port 3000)
    ↓
ragService.js → retrievalEngine.js
    ├─ 读 retrieval_index.json (根目录，⚠️ 已删除！)
    └─ 或 fallback 到 retrievalEngine 内部索引  ⚠️ 可能不是最新
    ↓
返回回答
```

## 五、重构方案建议

### 方案：统一 Single Source of Truth

#### 目标
1. **唯一数据源**: 所有文档索引只保存在 `文档库/indexes/` 目录
2. **唯一注册表**: 只有一个 registry.json，在 `文档库/` 目录
3. **唯一 RAG 服务**: 前台和后台共用同一个 RAG 检索实现
4. **健壮的 ID 管理**: 注册表 ID 和索引 ID 统一

#### 架构变更

```
┌──────────────────────────────────────┐
│         文档库 (Single Source)        │
│                                      │
│  文档库/                             │
│    ├─ config.json          ← 唯一配置│
│    ├─ registry.json        ← 唯一注册表│
│    ├─ indexes/                       │
│    │   ├─ retrieval_index.json  ← 唯一索引│
│    │   └─ embedding_cache.json  ← 缓存  │
│    └─ documents/            ← 源文件   │
│                                      │
│  (所有服务都读这个目录，不读写其他位置)  │
└──────────────────────────────────────┘
```

#### 具体改造项

| # | 改造项 | 优先级 | 说明 |
|---|--------|--------|------|
| 1 | 统一注册表位置到 `文档库/registry.json` | P0 | 删除其他两个 registry.json |
| 2 | 统一索引写入到 `文档库/indexes/retrieval_index.json` | P0 | 删除所有其他索引文件 |
| 3 | 修复 embedding 缓存加载逻辑 | P0 | chunks 已有 embedding，不需要单独缓存 |
| 4 | 修复 RAG 检索相似度问题 | P0 | 可能是 cosineSimilarity 或 embedding 问题 |
| 5 | 前台 RAG 改用共享的 retrieval_index.json | P0 | 前台也要读统一索引 |
| 6 | 注册表重启后数据恢复 | P1 | 确保 registry.json 路径正确 |
| 7 | 统一文档 ID 生成策略 | P1 | 注册时生成 ID，一直沿用到底 |
| 8 | 添加完整单元测试 | P2 | 上传→处理→检索→删除全流程测试 |
