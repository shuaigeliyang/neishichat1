## 一、项目背景与目标

### 1.1 现状问题

当前系统是针对"学生手册"单一文档设计的问答系统，存在以下问题：

| 问题 | 说明 |
|------|------|
| 单一文档限制 | 只能处理一个文档，新增政策需重建整个索引 |
| 无增量管理 | 无法单独添加、更新、删除某个政策文档 |
| 索引冲突 | 新旧政策内容混杂，无法区分来源 |
| 扩展性差 | 每增加一个政策文档就需要修改代码 |

### 1.2 改造目标

```
┌─────────────────────────────────────────────────────────────┐
│                      改造目标                                 │
├─────────────────────────────────────────────────────────────┤
│  ✓ 支持多政策文档管理（新增、替换、删除）                        │
│  ✓ 每个政策文档独立管理、独立索引                               │
│  ✓ 前台智能问答自动聚合所有已索引政策                           │
│  ✓ 增量更新，无需重建整个索引                                  │
│  ✓ 清晰的文档来源追踪                                          │
└─────────────────────────────────────────────────────────────┘
```

## 四、API接口设计

### 4.1 文档管理 API

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/documents` | 获取所有文档列表 |
| GET | `/api/documents/:documentId` | 获取单个文档详情 |
| POST | `/api/documents/upload` | 上传并注册新文档（新增） |
| POST | `/api/documents/:documentId/replace` | 替换文档（替换） |
| DELETE | `/api/documents/:documentId` | 删除文档（删除） |
| POST | `/api/documents/:documentId/process` | 开始处理文档 |
| GET | `/api/documents/:documentId/status` | 获取文档处理状态 |
| GET | `/api/documents/:documentId/chunks` | 获取文档分块详情 |
| GET | `/api/index/status` | 获取统一索引状态 |
| POST | `/api/index/rebuild` | 重建统一索引 |

### 4.2 详细接口说明

#### 4.2.1 新增政策文档

```
POST /api/documents/upload
Content-Type: multipart/form-data

参数：
- file: 文件（必填）
- name: 文档名称（可选，默认从文件名获取）
- description: 文档描述（可选）
- tags: 标签，逗号分隔（可选）

响应：
{
  "success": true,
  "data": {
    "documentId": "policy_003",
    "name": "学籍管理规定",
    "status": "pending",
    "directory": "文档提取/学籍管理规定"
  },
  "message": "文档上传成功，请开始处理"
}
```

#### 4.2.2 替换政策文档

```
POST /api/documents/:documentId/replace
Content-Type: multipart/form-data

参数：
- file: 新文件（必填）

响应：
{
  "success": true,
  "data": {
    "documentId": "policy_001",
    "oldStatus": "indexed",
    "newStatus": "pending"
  },
  "message": "文档已准备替换，请开始处理"
}
```

流程说明：
1. 保留原 documentId
2. 备份旧文件到 archive 目录
3. 上传新文件替换
4. 重置状态为 pending
5. 需要重新处理（向量化）

#### 4.2.3 删除政策文档

```
DELETE /api/documents/:documentId

响应：
{
  "success": true,
  "message": "文档已删除，相关索引已清理"
}
```

流程说明：
1. 从统一索引中移除该文档的所有 chunks
2. 从注册表中移除文档记录
3. 备份文件到 archive 目录
4. 更新统一索引文件

#### 4.2.4 处理文档（提取→分块→向量化→索引）

```
POST /api/documents/:documentId/process

响应：
{
  "success": true,
  "data": {
    "taskId": "task_xxx",
    "status": "processing"
  },
  "message": "文档处理已启动"
}
```

处理流程：
```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  提取   │ → │  分块   │ → │ 向量化  │ → │ 添加索引 │ → │ 更新注册表│
│ .docx   │    │ 智能分块 │    │ 生成向量│    │ 统一索引 │    │  status  │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

## 六、前台RAG系统改造

### 6.1 多文档检索流程

```
用户提问：奖学金怎么评定？
         │
         ▼
┌─────────────────────────────────────┐
│         检索引擎                      │
├─────────────────────────────────────┤
│  1. 加载统一索引                       │
│  2. 查询向量数据库                     │
│  3. 找出所有相关 chunk                 │
│  4. 按 documentId 分组                │
│  5. 标记来源文档                       │
└─────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────┐
│         回答生成                       │
├─────────────────────────────────────┤
│  回答：关于奖学金评定，根据：          │
│                                     │
│  📄 奖学金评定办法                    │
│     - 国家奖学金评定...（第3页）       │
│     - 校级奖学金...（第5页）          │
│                                     │
│  📄 学生手册                          │
│     - 奖学金基本条件...（第45页）      │
└─────────────────────────────────────┘
```

### 6.2 来源追踪

检索结果需要包含 documentId 和 documentName，以便：
1. 在回答中标注来源文档
2. 用户可以知道答案来自哪个政策
3. 支持按文档过滤检索结果

### 6.3 代码改造要点

```javascript
// retrievalEngine.js 改造
class RetrievalEngine {
    async retrieve(query, options = {}) {
        // 1. 向量检索
        const results = await this.vectorSearch(query, options.topK);

        // 2. 按 documentId 分组
        const groupedResults = this.groupByDocument(results);

        // 3. 标记来源
        return results.map(r => ({
            ...r,
            documentName: this.getDocumentName(r.documentId),
            sources: this.extractSources(r)
        }));
    }
}
```

## 九、注意事项

### 9.1 迁移现有数据

当前已有的"学生手册"数据需要迁移到新结构：
```
原位置：E:\外包\教育系统智能体\文档提取\2025年本科学生手册-定\
新结构：保持不变（已符合新规范）
```

需要操作：
1. 创建 registry.json
2. 创建 unified_index.json（从现有 retrieval_index.json 迁移）
3. 更新 chunks 结构添加 documentId

### 9.2 向量化成本

新增文档时需要生成新的向量，有API调用成本：
- embedding-3 模型：约 0.1元/1000次调用
- 一个100页文档约 500 个 chunks

### 9.3 文件命名规范

上传文件时会自动清理文件名：
- 移除特殊字符
- 限制长度（100字符）
- 用下划线替代空格
