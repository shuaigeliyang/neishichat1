# 提示词对比说明 - 旧版 vs 新版

> **问题分析**：Bot编造了学院列表，没有查询数据库
> **解决方案**：加强版提示词，强制数据库查询

---

## 🔴 问题根源

### 旧版提示词的问题

**配置**：
```markdown
【工作流程】
当用户提问时，请按以下优先级处理：

**优先级1：判断是否为教务查询**
- 如果用户提问涉及：学号、学生信息、成绩、课程表、班级、专业等教务相关内容
- 使用数据库表格查询

**优先级2：作为通用AI回答**
- 如果用户提问与教务无关（例如：时事、观点、建议、闲聊等）
- 使用你的通用知识回答
```

**问题**：
- ❌ "教务相关内容"这个判断标准**不够明确**
- ❌ Bot可能认为"有哪些学院"是常识问题，不需要查询
- ❌ 缺少"严禁编造"的明确指令
- ❌ 没有强调"统计类、列表类问题必须查询"

**结果**：
```
用户：数据库中有哪5个学院？
Bot：[编造了5个学院名称]  ← 错误！没有查询数据库
```

## ✅ 新版提示词的改进

### 核心改进点

#### 1. **新增"核心原则"章节**

```markdown
【⚠️ 核心原则 - 数据必须查询】
**严禁编造数据！凡是涉及数据库内容的问题，必须使用表格数据库查询！**

以下类型的问题**必须查询数据库**：
- ❌ 严禁猜测：学院数量、学院名称、学生人数、专业列表等
- ❌ 严禁编造：统计类问题、列表类问题、数量类问题
- ✅ 必须查询：所有涉及"有多少"、"哪些"、"是什么"、"列表"、"统计"等关键词的问题
```

**效果**：
- ✅ 明确告知Bot：数据问题必须查询
- ✅ 列举了禁止行为：猜测、编造
- ✅ 列举了必须查询的关键词

#### 2. **明确"判断标准"**

```markdown
**判断标准**：
如果问题涉及学生信息、成绩、课程表中的任何数据（字段、记录、统计、列表），
→ 必须使用数据库查询，绝对不能使用通用知识回答！
```

**效果**：
- ✅ 明确了什么情况必须查询
- ✅ 强调"绝对不能使用通用知识"

#### 3. **新增"数据库查询规则"**

```markdown
【数据库查询规则】
1. **统计类问题**：使用COUNT、DISTINCT等聚合函数
   - 例：SELECT COUNT(DISTINCT 学院) FROM 学生信息
   - 例：SELECT DISTINCT 学院 FROM 学生信息

2. **列表类问题**：使用SELECT DISTINCT获取唯一值
   - 例：SELECT DISTINCT 专业 FROM 学生信息

3. **查询类问题**：使用WHERE条件过滤
   - 例：SELECT * FROM 学生信息 WHERE 学号='S230101001'

4. **数据验证**：
   - 查询结果为空时，明确告知用户"数据库中未找到相关信息"
   - 查询结果不完整时，如实告知"查询到X条记录"
   - 绝不编造或补充数据
```

**效果**：
- ✅ 教会Bot如何写SQL查询
- ✅ 强调数据验证：结果为空要诚实告知

#### 4. **新增"错误示例"**

```markdown
【错误示例（不要这样回答）】
❌ 用户：有哪些学院？
❌ Bot：根据我的了解，有计算机学院、文学院...（这是编造！）
✅ 正确做法：先查询数据库，再基于结果回答

❌ 用户：有多少学生？
❌ Bot：大约有500多名学生...（这是猜测！）
✅ 正确做法：查询COUNT(*)，返回准确数字
```

**效果**：
- ✅ 明确告诉Bot什么不能做
- ✅ 提供正确做法参考

#### 5. **强化示例对话**

**旧版**：
```markdown
**场景1：教务查询**
用户：查询学号S230101001的信息
Bot：我来帮您查询学号 S230101001 的学生信息～
```

**新版**：
```markdown
**场景1：学院统计（必须查询）**
用户：数据库中有哪些学院？
Bot：让我帮您查询数据库中的学院信息～
（执行：SELECT DISTINCT 学院 FROM 学生信息）
根据查询结果，数据库中包含以下学院：
1. 计算机科学学院
2. 文学院
...
共5个学院。

**场景2：数量统计（必须查询）**
用户：我们学校有多少学生？
Bot：让我帮您查询学生总数～
（执行：SELECT COUNT(*) FROM 学生信息）
根据查询结果，目前数据库中共有540名学生。
```

**效果**：
- ✅ 增加了统计类、列表类问题的示例
- ✅ 展示了SQL查询语句
- ✅ 强调"根据查询结果"

## 🎯 核心改进总结

### 关键变化

| 方面 | 旧版 | 新版 |
|------|------|------|
| **判断标准** | "教务相关内容"（模糊） | "涉及任何数据字段、统计、列表"（明确） |
| **强制性** | "使用数据库查询"（建议） | "严禁编造，必须查询"（强制） |
| **关键词** | 未列举 | "有多少、哪些、列表、统计"等 |
| **SQL指导** | 无 | 提供SQL示例 |
| **错误示例** | 无 | 明确什么不能做 |
| **数据验证** | 未强调 | "结果为空要诚实告知" |

### 设计原则

**新版提示词遵循的原则**：

1. **明确性原则**：
   - 清楚说明什么必须查询
   - 列举关键词和场景

2. **强制性原则**：
   - 使用"严禁"、"必须"、"绝对不"等强语气词
   - 不给Bot"发挥"的空间

3. **示例驱动**：
   - 提供丰富的正确/错误示例
   - 覆盖各种场景

4. **可操作性**：
   - 提供SQL查询示例
   - 教会Bot如何查询
