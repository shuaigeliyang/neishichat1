### 问题1：Bot只能查教务，拒绝回答其他问题

**问题描述**：
```
用户：你怎么看待特朗普？
Bot：你好!作为"内江师院智能助手"，我的主要职责是为内江师范学院的
    学生和辅导员提供教务查询(如成绩、课表、个人信息)等服务。
    关于特朗普或政治人物等话题，不属于我的服务范围，我无法提供相关看法或评论。
```

**根本原因**：
- 人设配置过于限制性
- 明确拒绝了所有非教务问题
- 没有启用通用AI对话能力

**解决方案**：
- 重新设计人设配置为"双模式智能助手"
- 模式1：教务查询专家（保留原功能）
- 模式2：通用AI助手（新增能力）
- 智能判断用户意图，自动切换模式

**测试结果**：
```
用户：你怎么看待特朗普？
Bot：作为 AI 助手，我对政治人物保持中立态度。特朗普是美国第 45 任总统，
    他的政策和言行在美国乃至全球都引起了广泛讨论和不同看法。😊

    不过在内江师院，我们更关心的是如何为大家提供更好的教务服务和帮助大家学习！📚
    如果有查成绩、课表或者其他学校相关的问题，随时问我哦～
```

✅ 成功！既能回答日常问题，又能引导回教务服务

### 问题2：Bot编造数据，未查询数据库

**问题描述**：
```
用户：我让你说出5个学院的完整名字
Bot：非常抱歉!刚才的回答确实不完整，让您感到困惑了。根据我对数据库的完整查询，
    这5个学院的完整名字分别是:
    1.计算机科学学院
    2.文学院
    3.数学与信息科学学院
    4.外国语学院
    5.经济与管理学院
```

**问题分析**：
- Bot声称"查询"了数据库，实际可能是编造的
- 缺少"严禁编造数据"的明确指令
- 没有强调"统计类、列表类问题必须查询"

**解决方案**：
- 新增"核心原则：数据必须查询"
- 明确列举必须查询的关键词
- 添加SQL查询指导
- 新增错误示例说明

**关键改进**：
```markdown
【⚠️ 核心原则 - 数据必须查询】
**严禁编造数据！凡是涉及数据库内容的问题，必须使用表格数据库查询！**

以下类型的问题**必须查询数据库**：
- ❌ 严禁猜测：学院数量、学院名称、学生人数、专业列表等
- ❌ 严禁编造：统计类问题、列表类问题、数量类问题
- ✅ 必须查询：所有涉及"有多少"、"哪些"、"是什么"、"列表"、"统计"等关键词的问题
```

### 问题3：查询结果被截断

**问题描述**：
```
用户：数据库中有哪些学院？
Bot：根据我对学生信息表的查询结果，学生信息表中包含的学院是:
1.计算机科学与技术学院
2.软件工程学院
3.信息管理学院

用户：那电子工程学院和人工智能学院呢？
Bot：根据我对学生信息表的查询结果，电子工程学院和人工智能学院目前
    不在数据库的学生信息表中。
```

**问题分析**：
- 用户确认Coze数据库有完整数据（包含电子工程学院和人工智能学院）
- Bot只返回了3个学院，说明查询或回复被截断
- 可能是Coze平台自动添加了LIMIT限制

**解决方案**：

1. **强制禁止LIMIT**：
```markdown
❌ 严禁使用LIMIT限制行数
❌ 严禁使用TOP限制结果
✅ 必须使用SELECT DISTINCT获取所有值
```

2. **强制说明总数**：
```markdown
如果查询结果超过20项：
1. 说明总数量："共查询到X个[项目]"
2. 分批列出
3. 明确："以上是第1-20项，共X项"
4. 询问："需要查看完整列表吗？"
```

3. **调整模型设置**：
- 最大回复长度：从4096增加到8192或更高
- 生成多样性：使用"平衡模式"而非"创意模式"

## 🎯 关键技术要点

### 1. 人设配置设计

**核心原则**：
- 明确双重/多重能力定位
- 建立清晰的优先级机制
- 强调禁止行为（如：严禁编造、严禁截断）
- 提供丰富的示例对话

**结构设计**：
```
【核心能力】→ 明确定位
【核心原则】→ 强调重要规则
【工作流程】→ 优先级判断
【查询规范】→ SQL模式
【示例对话】→ 具体场景
【错误示例】→ 避免错误
```

**三个层次防护**：
1. **关键词列举**：明确什么必须查询
2. **SQL规范**：正确的查询模式
3. **回复格式**：如何完整展示

**关键语句**：
```markdown
- 严禁使用LIMIT限制行数
- 必须返回完整的查询结果
- 必须说明总数："共查询到X个"
```

## 💡 重要经验总结

### 1. 问题诊断思路

当Bot表现不符合预期时：
1. **先检查数据**：确认数据库是否完整
2. **再检查配置**：提示词是否清晰明确
3. **最后检查平台**：是否有技术限制（如回复长度）

### 2. 提示词设计原则

- **明确性**：清楚说明什么必须做、什么不能做
- **强制性**：使用"严禁"、"必须"等强语气词
- **示例驱动**：提供丰富的正确/错误示例
- **可操作性**：给出具体的SQL模式和回复格式

### 3. 渐进式优化

- 先解决核心功能（查询不截断）
- 再扩展能力（联网搜索、友好交流）
- 最后完善细节（情感支持、幽默表达）
