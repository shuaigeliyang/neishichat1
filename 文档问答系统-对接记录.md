## 📋 需求背景

### 核心需求
**用户（冰冰）的描述：**
> "这个智能体就是供内江师范学院学生和辅导员专用。他们可以通过问智能体得到和学生管理相关的内容和信息。这些内容必须是学校文件中给出来的，而正确的信息。"

**具体要求：**
- 🎯 **目标用户**：内江师范学院的学生和辅导员
- 📚 **核心功能**：提供学校官方文件中的准确信息查询
- 📄 **信息来源**：学校官方文档（必须基于文档，不能瞎编）
- ✅ **质量要求**：所有回答必须基于官方文件，确保准确性

**典型问题示例：**
- 重修管理办法
- 转专业管理办法
- 奖学金评定管理办法
- 其他学生管理规定

---

## 🏗️ 技术方案讨论

### 方案对比

#### ❌ 方案 B：独立开发文档问答系统
- 单独做一个文档问答系统
- 与现有数据库查询系统分离
- **被否决原因**：用户体验割裂，需要切换系统

#### ✅ 方案 A：在现有系统上增加 RAG 模块（已选定！）
- 统一的用户入口，一个聊天界面搞定所有问题
- 可以同时查询数据库和文档
- 智能路由：自动判断是数据库查询还是文档问答
- **优势**：用户体验流畅，技术架构优雅

```

    ↓

    ↓

```

**场景 2：文档问答**
```
学生："重修需要什么条件？"
    ↓
意图识别 → 文档问答
    ↓
检索《重修管理办法》
    ↓
返回：基于文档的准确答案（带引用）
```

**场景 3：混合查询**
```
学生："我挂了3门课，还能申请奖学金吗？"
    ↓
意图识别 → 混合查询
    ↓
1. 查询数据库：挂科情况
2. 检索文档：奖学金评定办法
    ↓
智能合并回答
```

## 🔧 技术选型

### 部署方式：本地部署（已确定！）

**选择理由：**
1. ✅ **数据安全**：学校官方文件不外泄，完全自主控制
2. ✅ **深度集成**：和现有系统无缝整合
3. ✅ **完全定制**：功能随需求定制，不受平台限制
4. ✅ **长期划算**：一次投入，永久使用
5. ✅ **合规性**：符合学校正式系统的要求

### RAG 技术栈

**文档存储：**
- 使用现有的 MySQL 数据库
- 表结构：`school_documents`（已设计）
- 字段：文档标题、内容、类型、版本、生效日期等

**向量化方案：**
- Embedding API：OpenAI Embeddings / 其他国产embedding服务
- 向量存储：MySQL + 向量字段（轻量级方案）
- **不需要复杂的向量数据库！**

**检索算法：**
- 语义相似度搜索
- 余弦相似度计算
- Top-K 相关片段检索

**问答生成：**
- 继续使用智谱 AI（保持一致性）
- 基于检索到的文档片段生成回答
- 引用来源，确保可追溯性

## 💬 讨论记录摘要

### 关键决策点

1. **部署方式**：本地部署（而非 Coze 等平台）
   - 原因：数据安全、深度集成、合规性

2. **技术方案**：在现有系统上增加 RAG 模块
   - 原因：统一用户体验，技术架构优雅

3. **技术栈**：MySQL + Embedding API + 智谱 AI
   - 原因：轻量级、易集成、成本低

### 重要共识

- 所有回答必须基于学校官方文档
- 确保信息的准确性和可追溯性
- 优先考虑数据安全和合规性
- 保持用户体验的流畅性
